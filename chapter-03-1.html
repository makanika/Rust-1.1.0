Chapter 3.2: The Router Configuration, Uncut

The Lifecycle of a `String`

This is a pure, practical lab. We have one goal: to follow a single piece of router configuration data from its birth in memory to its final cleanup. We will see how Ownership and Borrowing are not abstract rules, but a practical system for safely managing this data's lifecycle. Every example builds on the last.

1. The Birth: Creating the Initial Configuration

Our story begins in the `main` function. As a network engineer, your first step is to create a base configuration. We'll start with just a hostname.

fn main() {
    let mut config = String::from("hostname R1");
}

Let's break down exactly what happens in the machine when this single line of code runs.

Step-by-Step Memory Allocation

  1. Why `String`? The text `"hostname R1"` could change. We might add more lines. Because its size is not fixed and known at compile time, it must be stored on the Heap. The `String` type is Rust's standard tool for managing heap-allocated text.
  2. Allocation Request: The `String::from()` function makes a request to the memory allocator: "I need space on the Heap for 11 bytes of text."
  3. Heap Allocation: The allocator finds an empty spot, reserves it, and stores the bytes for "hostname R1" there. Let's say it's at memory address `0x9e2b0`.
  4. Stack Variable: Now, the variable `config` is created on the Stack. It does not hold the text. It holds a "fat pointer" containing three pieces of fixed-size information:
    • Pointer (`ptr`): The address of the data on the Heap (`0x9e2b0`).
    • Length (`len`): How many bytes of memory are currently being used (`11`).
    • Capacity (`cap`): How many bytes the allocator reserved on the Heap. For a new string, this is usually equal to the length (`11`).
  5. Ownership: At this moment, the variable `config` is declared the owner of the heap memory at `0x9e2b0`. It is now responsible for this data.
STACK
name: config
ptr: 0x9e2b0
len: 11
cap: 11
HEAP
addr: 0x9e2b0
val: "hostname R1"

2. The Audit: Immutable Borrows (&)

Now, a separate process—an auditor—needs to read this configuration to check its length. We don't want the auditor to take ownership; `main` is still working on the config. So, we lend them read-only access. This is an immutable borrow.

fn main() {
    let mut config = String::from("hostname R1");
    audit_config(&config); // Note the ampersand `&`
}

fn audit_config(config_ref: &String) {
    println!("Auditing... config length is {}.", config_ref.len());
}

What is a Reference, Physically?

A reference is a pointer. When we call `audit_config(&config)`, we are not passing the `String` struct. We are creating a new variable on the stack for `audit_config` called `config_ref`, and its value is simply the memory address of the original `config` variable. It's a "reference" to the original.

STACK
// audit_config frame
name: config_ref
ptr: 0x7ffc... (addr of config)

// main frame
name: config
ptr: 0x9e2b0
len: 11
cap: 11
HEAP
addr: 0x9e2b0
val: "hostname R1"

The Compiler as Security Guard

What if the auditor tries to change the config? Let's try it.

fn audit_config(config_ref: &String) {
    config_ref.push_str(" // AUDITED"); // Will this work?
}

If you try to compile this, you get a clear error:

error[E0596]: cannot borrow `*config_ref` as mutable, as it is behind a `&` reference
 --> src/main.rs:7:5
  |
6 | fn audit_config(config_ref: &String) {
  |                           ------- help: consider changing this to be a mutable reference: `&mut String`
7 |     config_ref.push_str(" // AUDITED");
  |     ^^^^^^^^^^ `config_ref` is a `&` reference, so the data it refers to cannot be borrowed as mutable

The compiler is acting as a security guard. It sees that `config_ref` is a read-only pass (`&String`) and denies the attempt to write. It even suggests the fix: if you want to write, you need a different kind of pass (`&mut String`).

3. The Reconfiguration: Mutable Borrows (&mut)

Next, an administrator needs to add an interface to the config. This requires write access. For this, we use a mutable borrow (`&mut`). This grants exclusive access.

fn main() {
    let mut config = String::from("hostname R1");
    add_interface(&mut config); // Note the `&mut`
}

fn add_interface(config_ref: &mut String) {
    config_ref.push_str("\ninterface Gi0/1");
}

The "One Writer" Rule

A mutable borrow is a powerful, exclusive lock. While a mutable borrow is active, the compiler will not allow *any* other access to the original data, not even for reading. This is the heart of Rust's "fearless concurrency." It prevents data races at compile time.

Let's prove it by trying to have an auditor and an admin work at the same time.

fn main() {
    let mut config = String::from("hostname R1");
    let admin_access = &mut config; // Mutable borrow starts here
    let auditor_access = &config;  // Immutable borrow tries to start here

    // ... use admin_access and auditor_access
}
error[E0502]: cannot borrow `config` as immutable because it is also borrowed as mutable
 --> src/main.rs:4:26
  |
3 |     let admin_access = &mut config;
  |                        ----------- mutable borrow occurs here
4 |     let auditor_access = &config;
  |                          ^^^^^^^ immutable borrow occurs here

The compiler stops you cold. It explains that you can't have a read-only `auditor_access` while the write-access `admin_access` exists. This rule prevents an auditor from reading a half-written configuration, which could lead to disastrous errors in a real system.

4. The Deployment: The Move

The configuration is finalized. It's time to hand it off to the deployment system. We don't want `main` to be able to change it after this point. We need to transfer full responsibility. We do this by passing the variable by value, which causes a move.

fn main() {
    let mut config = String::from("hostname R1\ninterface Gi0/1");
    deploy_config(config); // No `&`. We are moving ownership.
    
    // `config` is now invalid here!
}

fn deploy_config(final_config: String) {
    println!("Deploying config: {}", final_config);
    // `final_config` is now the owner.
}

Visualizing the Move

This is the most critical transition to understand.

  1. Before the call: `main`'s `config` variable on the Stack owns the Heap data.
  2. During the call: A new stack frame is created for `deploy_config`. The content of the `config` variable (the ptr, len, cap) is copied to the `final_config` variable.
  3. The Invalidation: To enforce the "one owner" rule, the compiler immediately marks the original `config` variable in `main` as uninitialized and invalid. It cannot be used again.
STACK
// deploy_config frame
name: final_config
ptr: 0x9e2b0
len: 28
cap: 28

// main frame
name: config
ptr: 0x9e2b0
len: 28
cap: 28
HEAP
addr: 0x9e2b0
val: "hostname R1\n..."

If you try to use `config` in `main` after the call, you get the familiar error:

error[E0382]: borrow of moved value: `config`

5. The Cleanup: The `drop`

Our story ends when the owner of the data goes out of scope. In our example, this happens when the `deploy_config` function finishes.

Automatic Deallocation

Because `final_config` is the sole owner, when it goes out of scope at the end of the function, Rust automatically inserts a call to a special function called `drop`. This function's job is to return the memory on the Heap to the allocator. The rack is now officially decommissioned and can be used by someone else.

This is the beauty of the system. Cleanup is deterministic and automatic. It happens as soon as the owner is done, not at some random time when a garbage collector feels like running. For embedded systems, this predictability is a priceless feature.

Final Thoughts and a Preamble to Chapter 4

You have now seen the entire lifecycle of a piece of dynamic data, from its allocation on the Heap to its cleanup when the owner is dropped. You have a physical, mechanical intuition for what `move`, `borrow`, and `mutable borrow` actually mean in memory. This is the hardest part of learning Rust, and you have just tackled it head-on.

In our case study, we used a simple `String` to represent our entire router configuration. This is unrealistic. A real configuration is a complex collection of different data: hostnames, interface settings, access control lists, routing protocols, and more.
