Chapter 4: Blueprints for Data

Structs and Enums

We have mastered how data is owned and passed around. Now, we learn to design the data itself. This chapter introduces Rust's two primary tools for creating custom data types: `struct` for grouping related data, and `enum` for representing a choice among possibilities. We will use them to build a detailed, realistic model of our network router configuration.

Structuring Your Data with `struct`

A `struct`, short for structure, is a custom data type that lets you name and package together related values. It's like a blueprint for a concept. In Python, you might use a dictionary or a simple class for this. In Rust, a `struct` is the clear, type-safe way to do it.

Case Study: Modeling a Network Interface

An interface on a router isn't just one value; it's a collection of properties. We can model this with a struct.


struct NetworkInterface {
    name: String,
    ip_address: String,
    is_enabled: bool,
    mtu: u16,
}

fn main() {
    // We create an "instance" of the struct
    let gig_ethernet_0_1 = NetworkInterface {
        name: String::from("GigabitEthernet0/1"),
        ip_address: String::from("192.168.1.1"),
        is_enabled: true,
        mtu: 1500,
    };

    // We can access the fields using dot notation
    println!("Interface {} is up with IP {}", 
        gig_ethernet_0_1.name, 
        gig_ethernet_0_1.ip_address
    );
}

Structs and Ownership

The rules of ownership apply to structs as a whole. Our `NetworkInterface` struct contains `String` fields, which own data on the Heap. This means that when we assign an instance of `NetworkInterface` to another variable, the entire struct is moved, just like a `String` was in the previous chapter.

Structs in Memory

A struct's fields are stored contiguously in memory, in the order you declare them. Let's visualize our `gig_ethernet_0_1` variable on the stack:

// On the Stack for `main` function:
name: gig_ethernet_0_1
  .name (ptr): 0xABC1
  .name (len): 18
  .name (cap): 18
  .ip_address (ptr): 0xDEF2
  .ip_address (len): 11
  .ip_address (cap): 11
  .is_enabled: true
  .mtu: 1500

// On the Heap:
addr: 0xABC1 "GigabitEthernet0/1"
addr: 0xDEF2 "192.168.1.1"

The stack part of the struct contains the pointers to the heap data, alongside the fixed-size values like the boolean and the `u16`.

Giving Structs Behavior with `impl`

Data is more useful when it has associated behavior. In Rust, we define methods for a struct in an `impl` block (short for "implementation").

// (struct definition from before)
struct NetworkInterface {
    name: String,
    ip_address: String,
    is_enabled: bool,
    mtu: u16,
}

impl NetworkInterface {
    // This is a "method". Its first parameter is always `self`.
    // `&self` is an immutable borrow of the struct instance.
    fn get_summary(&self) -> String {
        format!("{}: {} (status: {})", 
            self.name, 
            self.ip_address, 
            if self.is_enabled { "up" } else { "down" }
        )
    }
    
    // `&mut self` is a mutable borrow, allowing us to change the instance.
    fn disable(&mut self) {
        self.is_enabled = false;
        println!("Interface {} has been disabled.", self.name);
    }
}

fn main() {
    let mut interface1 = NetworkInterface {
        name: String::from("vlan100"),
        ip_address: String::from("10.10.100.1"),
        is_enabled: true,
        mtu: 1500,
    };
    
    println!("{}", interface1.get_summary());
    interface1.disable();
    println!("{}", interface1.get_summary());
}

The use of `&self` and `&mut self` is not arbitrary. It's a direct application of the borrowing rules we learned in Chapter 3. The compiler enforces them here just as it did with standalone functions.

Enums: A Value Can Be One of a Few Things

Sometimes, a value can only be one of a small, finite set of possibilities. For this, we use an `enum`, short for "enumeration". This is far safer than using magic strings or numbers.

Case Study: Modeling Interface Status and IP Protocols

An interface's status can only be in a few states. An IP packet is either TCP or UDP. An IP address is either v4 or v6. Enums are perfect for this.


enum InterfaceStatus {
    Up,
    Down,
    AdminDown,
}

enum IpProtocol {
    Tcp,
    Udp,
}

// Enums can also hold data. This is a very powerful feature.
enum IpAddress {
    V4(u8, u8, u8, u8),
    V6(String),
}

fn main() {
    let status = InterfaceStatus::Up;
    let protocol = IpProtocol::Tcp;
    let home_ip = IpAddress::V4(127, 0, 0, 1);
    let loopback_ip = IpAddress::V6(String::from("::1"));
}

By using enums, we make our program's domain explicit. A function that expects an `InterfaceStatus` cannot accidentally be passed a `u8` with the value `5`. The compiler guarantees it.

The `match` Operator: Rust's Switch Statement

So how do we use a value that's an enum? The primary tool is the `match` control flow operator. It allows you to compare a value against a series of patterns and then execute code based on which pattern matches.

The most important feature of `match` is that it is exhaustive. You must provide a branch for every possible variant of the enum. The compiler will stop you if you forget one, eliminating a huge source of bugs.


// (Assuming the IpAddress enum from before)
fn route_packet(ip: IpAddress) {
    match ip {
        IpAddress::V4(octet1, octet2, octet3, octet4) => {
            println!("Routing IPv4 packet to {}.{}.{}.{}", 
                octet1, octet2, octet3, octet4);
        }
        IpAddress::V6(address) => {
            println!("Routing IPv6 packet to {}", address);
        }
    }
}

In the `V4` branch, `match` also de-structures the enum, binding the inner values to new variables we can use. If we had forgotten the `V6` branch, this code would not compile.

Chapter Project: An Advanced Router Model

Let's combine everything we've learned to build a sophisticated model of a router and its interfaces. This project will use structs, enums, methods, and the `match` operator to create a robust and expressive piece of code.

Your Task: Build the Router Model

  1. Create a new binary crate: `cargo new router_model`.
  2. Replace the code in `src/main.rs` with the complete code below.
  3. Read every comment to understand how the pieces fit together.
  4. Run the project with `cargo run` and analyze the output.

Final Code: `src/main.rs`


#[derive(Debug)]
enum InterfaceStatus {
    Up,
    Down,
    AdminDown,
}

#[derive(Debug)]
enum IpAddress {
    V4(u8, u8, u8, u8),
    V6(String),
}

// Our blueprint for a single network interface
#[derive(Debug)]
struct Interface {
    name: String,
    status: InterfaceStatus,
    ip: Option<IpAddress>, // An interface might not have an IP!
}

impl Interface {
    fn print_status(&self) {
        // We use `match` to handle the different status variants
        let status_str = match self.status {
            InterfaceStatus::Up => "Up",
            InterfaceStatus::Down => "Down",
            InterfaceStatus::AdminDown => "Administratively Down",
        };

        // We use `match` again to handle the Option for the IP address
        let ip_str = match &self.ip {
            Some(IpAddress::V4(o1, o2, o3, o4)) => format!("{}.{}.{}.{}", o1, o2, o3, o4),
            Some(IpAddress::V6(addr)) => addr.clone(),
            None => String::from("Unassigned"),
        };
        
        println!(
            "  Interface: {}, Status: {}, IP: {}",
            self.name, status_str, ip_str
        );
    }
}

fn main() {
    let gig0_1 = Interface {
        name: String::from("GigabitEthernet0/1"),
        status: InterfaceStatus::Up,
        ip: Some(IpAddress::V4(192, 168, 1, 1)),
    };
    
    let gig0_2 = Interface {
        name: String::from("GigabitEthernet0/2"),
        status: InterfaceStatus::AdminDown,
        ip: None, // This interface has no IP address assigned
    };

    println!("Router Status Report:");
    gig0_1.print_status();
    gig0_2.print_status();
}

Notice the `Option`. This is an enum from the standard library (`Some` or `None`) that lets us safely represent an interface that may or may not have an IP address. The compiler forces us to handle the `None` case, preventing null pointer errors.

Final Thoughts and a Preamble to Chapter 5

You have just taken a massive step forward. You are no longer limited to the basic types provided by the language. You can now create your own rich, expressive, and type-safe blueprints that precisely model the domain you are working in. You've seen how `struct` and `enum` are not just conveniences, but powerful tools for correctness that the compiler helps you enforce.

However, our router model is still simplistic. A real router doesn't have just two interfaces; it has a list of them. An access control list isn't a single rule; it's a collection of rules. How do we manage these lists of data?

In Chapter 5, "Collections and Generics," we will explore Rust's standard library collections. We'll do a deep dive into the `Vec` (a growable list, like Python's `list`), the `HashMap` (like Python's `dict`), and see how Rust uses generics to create data structures that can hold any type of data in a safe and efficient way.
